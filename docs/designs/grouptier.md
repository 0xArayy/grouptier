## Vision

### 10x Check

GroupTier as a platform — any Telegram group, any decision type, any number of options — becomes the default group decision layer on Telegram. Public bot @grouptier_bot, open source, with a revenue model via Telegram Stars. The pairwise comparison + aggregation mechanic generalizes beyond food/games to product feedback, hiring decisions, design choices, content curation. One product that understands group preferences better than the group understands itself. The 12-month ideal: GroupTier is to group decisions what Google Forms is to individual surveys — but native, social, and alive.

### Platonic Ideal

A Telegram group makes a decision. One person types /startsession. Another types five options. The admin sends the link. For the next five minutes, everyone taps through comparisons — it feels like Tinder, takes 30 seconds each, and requires zero thought. The tier list appears. The group sees their collective ranking update in real time as each person submits. The winner gets announced in the chat. Someone shares their personal tier list. The group laughs, agrees, or argues. The decision is made.

That's the product. Every engineering and design decision should be measured against whether it makes that moment better.

## Accepted Scope (added to this plan)

### CP1 — Live vote count
- Trigger: fires on `POST /api/sessions/:id/results` (final results submission per voter — one edit per voter who completes all comparisons)
- Requires: `message_id` column on Session table (stored when bot sends the /vote message via CP5)
- **CP1 + CP5 interaction:** `editMessageText` must re-supply the `reply_markup` inline keyboard (CP5's vote button) or Telegram will remove it. Always pass `reply_markup: { inline_keyboard: [[{ text: "🗳️ Cast your vote →", url: miniAppUrl }]] }` on every edit.
- Error handling: `editMessageText` failures are swallowed silently — session proceeds; log the error but do not throw
- Bot edits message text to: `"🗳️ Voting open for [session name]!

2 of 4 voted"` (inline button preserved via reply_markup)
- **No "all voted" announcement.** Sessions never close. The live tier list IS the product — it forms after the first submission and updates with each new one, forever. The bot edit (CP1) is purely informational: "2 of 4 voted so far" with no completion trigger. Any user can open the Mini App at any time and vote; the group Borda ranking updates immediately.
- voter_ids is a running tally (added on first GET /session/:id with valid initData), not a completion condition.

### CP2 — Session names
- `/startsession [name]` — name is optional
- Default if omitted: `null`, displayed as "Untitled Session" in all bot messages
- Grammy command: `bot.command("startsession", (ctx) => { const name = ctx.match || "Untitled Session"; ... })`
- One additional `name TEXT` column on Session table

### CP4 — 'Share my picks' button
- BotFather prerequisite: enable inline mode via `/setinline` before deployment
- **UX dead-end awareness:** `switchInlineQuery()` closes the Mini App and opens the inline composer. The user cannot navigate back. Only show the share button AFTER the user has submitted their result server-side (POST /api/sessions/:id/results completed). If shown before submission, share closes the app with unsaved data.
- Frontend: `Telegram.WebApp.switchInlineQuery(sessionId + ":" + userId, ["groups"])` — pass IDs, not the formatted string (Telegram limits inline query length)
- Bot `on("inline_query")` handler: parse `sessionId:userId` from `ctx.inlineQuery.query`, fetch `UserResult` from DB, format and return `InlineQueryResultArticle`
- Shared message format: `"My GroupTier picks for [session name]:
S: [options]
A: [options]
B: [options]
C: [options]"`
- Auth: inline query handler uses `ctx.from.id` (the actual sender) to look up the UserResult — do NOT use a userId from the query string, which is spoofable. Query: `SELECT * FROM user_results WHERE session_id = :sid AND user_id = ctx.from.id`

### CP5 — Inline vote button
- `bot.api.sendMessage(chatId, text, { reply_markup: { inline_keyboard: [[{ text: "🗳️ Cast your vote →", url: miniAppUrl }]] } })`
- No additional BotFather config needed

### CP6 — Session close + winner announcement (added by /autoplan premise gate)
- New bot command: `/closesession` — any group member can trigger, session creator preferred
- When triggered: bot sends new message to group: `"🏆 [Session Name] winner: [Top Option]!\n\nFull group ranking:\n1. [Option] — [Borda score] pts\n2. [Option] — [Borda score] pts\n..."`
- Data model: add `status TEXT DEFAULT 'open'` column to sessions table (`'open'` | `'closed'`)
- When session is closed: `GET /api/sessions/:id` returns `status: 'closed'`; Mini App shows results-only view (no new comparisons accepted)
- `POST /api/sessions/:id/results` returns 403 if session is closed
- Bot queries `GET /api/sessions/:id` on `/closesession` to fetch Borda ranking, formats winner message
- Borda winner = option with highest aggregate score at time of close
- Winner announcement is a new `sendMessage` (not an edit of CP1's message)
- This resolves the internal inconsistency between the Platonic Ideal ("winner announced in chat") and CP1 ("no announcement")

The single-service Railway deployment is right for the hackathon. One service = one URL = zero CORS risk. The accepted cherry-picks add:
- Two DB columns: `sessions.message_id`, `sessions.name`
- One bot API call per submission: `editMessageText`
- One inline query handler route
- One BotFather inline mode config step (CP4)

Frontend served by Fastify: `fastify.register(fastifyStatic, { root: path.join(__dirname, '../frontend/dist') })`

None of the cherry-picks change the core architecture. The data model remains clean for future WebSocket + Redis migration.

### Section 1: Architecture Review

**Dependency Graph:**
```
Telegram Group Chat
  ↓ /startsession [name]   /addoption [text]   /vote   /closesession
grammy bot (bot.ts)
  ↓ writes → PostgreSQL
  ├── sessions(id, name, chat_id, message_id, status)
  ├── options(id, session_id, text)
  ├── comparisons(id, session_id, option_a, option_b, winner, user_id)
  ├── user_results(id, session_id, user_id, ranked_list JSONB)
  └── session_voters(session_id, user_id) ← ON CONFLICT DO NOTHING
  ↓ CP5: sendMessage with inline keyboard (url → miniAppUrl?session_id=X)
  ↓ CP1: editMessageText on every POST /results (re-supply reply_markup)
  ↓ CP6: sendMessage winner announcement on /closesession
Fastify REST API (server/)
  GET  /api/sessions/:id  → {options, comparisons[], borda_ranking, status}
  POST /api/sessions/:id/results → save ranked_list; fire CP1 bot edit
  GET  /health            → 200 (Railway health check)
  ↓ serves static
React Mini App (frontend/dist/)
  /compare  → pairwise cards (client-side bracket, POST /results at end)
  /results  → personal TierList (S=red, A=orange, B=yellow, C=green)
  /live     → group Borda ranking (3s poll on GET /session/:id)
  ↑ launched via inline keyboard URL
  ↑ CP4: switchInlineQuery() after results submit → inline composer
```

**Data flow — POST /api/sessions/:id/results:**
- Happy path: client sends `{ranked_list: [...], initData: "..."}` → HMAC validate → upsert user_results → ON CONFLICT DO NOTHING in session_voters → return updated Borda ranking → fire editMessageText (CP1)
- Nil path: initData missing → 401
- Empty path: ranked_list is [] → 400 (validate non-empty)
- Error path: editMessageText fails → swallow + log, return 200 to client

### Section 3: Security & Threat Model

- **CP4 spoofing:** `ctx.from.id` used for auth (not query string userId) — PASS
- **initData HMAC:** server-side validation required — tracked in eng task T2
- **Rate limiting on POST /results:** closed by HMAC validation (can't forge initData)
- **Session isolation:** no uniqueness constraint on (chat_id, status='open') — MEDIUM gap; multiple open sessions per group is confusing. Fix: add unique index or return most-recent-active session.

### Section 4: Data Flow Edge Cases

- **N=2:** 1 comparison → winner = S tier, loser = A tier (define explicitly: rounds determine tier, not just win/loss count)
- **BYE UX:** option auto-advances after 1 second (from test plan); user sees countdown animation or auto-flip — **this must be specified**. BYE option gets tier of the round it EVENTUALLY LOSES IN (not a gift tier)
- **Tier assignment formula:** S = final round winner, A = final round loser, B = semifinal losers, C = remainder. Add this to plan explicitly.
- **Two users submit simultaneously:** session_voters ON CONFLICT DO NOTHING prevents double-count — PASS
- **User re-submits after close:** POST /results returns 403 — PASS (CP6 spec)

**Architecture note — auth boundary:**
- Bot commands trust Telegram server-side auth (`ctx.from.id` is guaranteed by Telegram). No HMAC needed.
